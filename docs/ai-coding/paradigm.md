# AI 编程范式与规范

> 从个人尝鲜到团队落地，AI 编程不是"让 AI 写代码"那么简单。
> 需要建立范式、规范和流程，才能真正实现效率跃升。

---

## 一、AI 编程的核心范式

### 1.1 三种工作范式 🔴🔴

| 范式 | 模式 | 人机分工 | 适用场景 | 典型工具 |
|------|------|---------|---------|---------|
| **Copilot 模式** 🔴🔴🔴 | AI 辅助补全 | 人主导写，AI 补全建议 | 日常编码、函数实现 | GitHub Copilot、Cursor Tab |
| **Agent 模式** 🔴🔴 | AI 自主执行 | 人给任务，AI 执行+反馈 | 功能开发、Bug 修复 | Cursor Agent、Codex、Claude Code |
| **对话模式** 🔴 | 问答式交互 | 人提问，AI 回答 | 架构设计、知识查询 | ChatGPT、Claude、通义 |

**关键认知** 🔴🔴：AI 不是替代程序员，而是将程序员的角色从"代码编写者"转变为"代码审查者+任务编排者"。

### 1.2 AI 编程的三大原则 🔴🔴🔴

**原则一：人始终在环（Human-in-the-Loop）**

- AI 生成的内容必须经过人审查才能合入
- 人负责：需求理解、架构决策、代码审查、最终确认
- AI 负责：代码生成、测试编写、文档补充、重复性工作
- ❌ 危险做法：AI 生成代码直接提交
- ✅ 正确做法：AI 生成 → 人工审查 → 测试通过 → 提交

**原则二：上下文决定质量（Context is King）**

- AI 输出质量 = 输入上下文质量
- 好的上下文：清晰的 Prompt、项目规范文件、示例代码、约束条件
- 差的上下文：模糊的需求、缺少约束、没有项目上下文
- 🔴 投入时间优化上下文，比花时间修改 AI 输出更高效

**原则三：可验证优先（Verifiable First）**

- 给 AI 的任务必须是可验证的：能跑测试、能看效果、能量化
- 不可验证的任务 = 不可控的风险
- ✅ "实现用户登录接口，包含单元测试" — 可验证
- ❌ "优化代码质量" — 不可验证（太模糊）

## 二、AI 编程规范

### 2.1 Prompt 规范 🔴🔴🔴

> 写好 Prompt 是 AI 编程最核心的技能

#### Prompt 五要素法 🔴🔴

```
1. 角色（Role）     — 你是谁
2. 任务（Task）     — 你要做什么
3. 上下文（Context）— 背景信息和约束
4. 示例（Example）  — 期望的输入输出格式
5. 格式（Format）   — 输出形式要求
```

**示例**：

```
【角色】你是一名 Java 后端开发工程师，熟悉 Spring Boot 和 MySQL。
【任务】实现一个用户注册接口 POST /api/users/register
【上下文】
- 项目使用 Spring Boot 3.2 + MyBatis-Plus + MySQL 8
- 用户表字段：id, username, password(BCrypt加密), email, created_at
- 需要参数校验：username 4-20位，email 格式合法，password 8位以上含大小写数字
- 遵循项目现有代码风格：Controller 不含业务逻辑，Service 层处理业务
【示例】参考现有 LoginController 的写法
【格式】输出完整的 Controller + Service + DTO 代码，并附带单元测试
```

#### Prompt 分类模板 🔴

| 场景 | 模板要点 |
|------|---------|
| 功能开发 | 需求描述 + 技术栈 + 代码风格 + 参考文件 + 测试要求 |
| Bug 修复 | 错误现象 + 复现步骤 + 相关代码 + 日志/堆栈 + 已尝试的方案 |
| 代码重构 | 当前代码 + 重构目标 + 约束条件 + 不能改变的行为 |
| 代码审查 | 待审代码 + 审查维度（安全/性能/可维护性）+ 项目规范 |
| 文档编写 | 代码/接口 + 文档类型 + 受众 + 格式要求 |

### 2.2 代码生成规范 🔴🔴

#### 生成代码的审查清单 🔴🔴🔴

每次 AI 生成代码后，必须逐项检查：

| 维度 | 检查项 |
|------|--------|
| **正确性** | 逻辑是否正确？边界条件是否处理？空值/null 是否处理？ |
| **安全性** | 是否有 SQL 注入风险？敏感信息是否硬编码？权限校验是否完整？ |
| **性能** | 是否有 N+1 查询？是否有不必要的循环？大数据量是否分页？ |
| **规范性** | 是否符合团队代码风格？命名是否规范？注释是否充分？ |
| **可维护性** | 是否过度设计？是否可测试？是否与现有代码风格一致？ |
| **依赖** | 是否引入了不必要的依赖？是否使用了未经审批的第三方库？ |

#### AI 代码标注规范 🔴🔴

所有 AI 生成或辅助生成的代码，必须标注来源：

```java
// [AI-Generated] 由 Cursor Agent 生成，人工审查通过
// [AI-Assisted]  核心逻辑由人工编写，AI 辅助补全
// [AI-Reviewed]  人工编写，AI 辅助审查
```

**标注原则**：
- AI 生成超过 50% 的函数/类 → `[AI-Generated]`
- 人工写核心逻辑，AI 补全细节 → `[AI-Assisted]`
- 人工编写，AI 做审查建议 → `[AI-Reviewed]`

### 2.3 上下文管理规范 🔴🔴🔴

#### 项目级上下文文件 🔴🔴🔴

| 文件 | 作用 | 放置位置 |
|------|------|---------|
| `.cursorrules` / `CLAUDE.md` | 项目规范、技术栈、代码风格 | 项目根目录 |
| `AGENTS.md` | AI Agent 的行为指引 | 项目根目录 |
| `.ai/rules/` | 细分的规则文件（安全、风格等） | 项目根目录 |

**`.cursorrules` 模板** 🔴🔴：

```
# 项目规范

## 技术栈
- 后端：Java 17 + Spring Boot 3.2 + MyBatis-Plus
- 前端：React 18 + TypeScript + Ant Design
- 数据库：MySQL 8 + Redis 7
- 构建：Maven + Docker

## 代码风格
- 使用 4 空格缩进
- 类名 PascalCase，方法名 camelCase，常量 UPPER_SNAKE_CASE
- 每个 public 方法必须有 Javadoc
- Controller 方法不超过 20 行，复杂逻辑放 Service

## 约束
- 禁止使用 System.out.println，必须用 SLF4J
- 数据库操作必须使用 MyBatis-Plus，禁止裸写 SQL
- 所有 API 必须有参数校验
- 密码必须 BCrypt 加密，禁止明文存储
- 分页查询必须使用 Page 对象

## 测试要求
- Service 层必须有单元测试
- 覆盖率要求：核心逻辑 > 80%
- 测试命名：test_{方法名}_{场景}_{预期结果}
```

#### 会话级上下文管理 🔴

| 原则 | 说明 |
|------|------|
| 一次一个任务 | 不要在一个会话中混合多个不相关任务 |
| 先给上下文 | 让 AI 读相关文件再开始任务 |
| 渐进式推进 | 先设计后实现，先接口后实现，先主干后细节 |
| 及时总结 | 任务完成后让 AI 总结变更，便于 review |

### 2.4 版本控制规范 🔴🔴

| 规范 | 说明 |
|------|------|
| commit 标注 | AI 生成代码的 commit 需标注：`feat: 用户注册接口 [AI-Assisted]` |
| 小步提交 | AI 生成的代码应小步提交，避免大量代码一次合入 |
| PR 描述 | 标注 AI 参与程度、审查要点、需要人工重点审查的部分 |
| 分支策略 | AI 辅助开发使用 feature 分支，禁止直接在 main 分支上操作 |

### 2.5 安全规范 🔴🔴🔴

| 红线 | 说明 |
|------|------|
| 🚫 禁止将密钥/Token 写入代码 | AI 生成的代码可能硬编码敏感信息 |
| 🚫 禁止将生产数据输入 AI | 脱敏后才能用于 Prompt |
| 🚫 禁止跳过代码审查 | AI 代码必须经过人工安全审查 |
| 🚫 禁止使用未审批的第三方库 | AI 可能引入有安全漏洞的依赖 |
| ✅ 定期审计 AI 生成代码 | 安全部件（认证、加密、权限）必须人工重点审查 |

## 三、AI 编程效率度量

### 效率指标 🔴

| 指标 | 度量方式 | 基线 | AI 辅助后预期 |
|------|---------|------|-------------|
| 代码产出 | 行/天或功能点/天 | 行业平均 | 提升 30-50% |
| Bug 密度 | Bug 数/KLOC | 5-10 | 降低 20-40% |
| 首次通过率 | 代码审查一次通过率 | 60-70% | 70-80% |
| 文档完整率 | 有文档的接口占比 | 30-40% | 80%+ |
| 测试覆盖率 | 单测覆盖率 | 40-60% | 70-80% |

### 度量原则 🔴

- 不以"AI 生成了多少代码"为唯一指标
- 关注"从需求到上线"的整体效率
- 代码质量（可维护性）比代码量更重要
- 团队成员的 AI 使用率应逐步提升
