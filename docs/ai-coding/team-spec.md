# 团队 AI 编程规范

> 不同规模的团队，AI 编程的痛点和规范完全不同。
> 小团队怕混乱，中团队怕冲突，大团队怕失控。

---

## 一、通用规范（所有团队必遵）

### 1.1 底线红线 🔴🔴🔴

| 红线 | 原因 |
|------|------|
| AI 生成代码必须经人工审查 | AI 可能引入逻辑错误、安全漏洞 |
| 敏感数据禁止输入 AI | 数据泄露风险 |
| 禁止 AI 直接提交到主分支 | 质量不可控 |
| 第三方库必须审批 | 安全和许可证风险 |
| 关键模块（认证/支付/加密）AI 仅辅助 | 核心安全逻辑必须人工主导 |

### 1.2 工具选型建议 🔴

| 类别 | 推荐工具 | 适用场景 |
|------|---------|---------|
| 代码补全 | GitHub Copilot | 日常编码 |
| AI IDE | Cursor / Windsurf | 全流程 AI 辅助 |
| Agent 模式 | Claude Code / Codex CLI | 自主任务执行 |
| 代码审查 | CodeRabbit / PR-Agent | PR 自动审查 |
| 知识问答 | ChatGPT / Claude | 技术咨询、方案设计 |

### 1.3 新人入职 AI 编程培训清单 🔴

- [ ] AI 编程范式培训（Copilot/Agent/对话三种模式）
- [ ] Prompt 编写规范培训
- [ ] 项目 `.cursorrules` / `CLAUDE.md` 文件解读
- [ ] AI 代码审查清单实操
- [ ] 安全红线考试（必须通过）

## 二、小型团队（2～3 人）🔴🔴

> 特点：沟通成本低、迭代快、但容易各自为战

### 2.1 核心风险 🔴

| 风险 | 表现 | 后果 |
|------|------|------|
| 风格不统一 | A 用 Copilot 生成、B 用 Cursor Agent 生成，代码风格差异大 | 维护困难 |
| 上下文断裂 | 每人自己维护 AI 上下文，没有共享 | AI 输出不一致 |
| 审查缺失 | "就两个人，互相看看就行" → 实际没人看 | 质量失控 |
| 过度依赖 | "AI 写的应该没问题" | Bug 累积 |

### 2.2 规范 🔴🔴🔴

#### 上下文规范

```
✅ 必须做：
1. 项目根目录维护统一的 .cursorrules / CLAUDE.md
2. 所有人使用相同的 AI 工具和配置
3. 每次迭代前同步更新上下文文件

❌ 禁止做：
1. 每个人维护自己的 Prompt 模板而不共享
2. 不更新 .cursorrules 就开始新功能开发
```

#### 流程规范

```
开发流程：
1. 早上站会同步：今天 AI 帮你做什么？遇到什么问题？
2. 功能开发：feature 分支 + AI 辅助 + 小步提交
3. 代码审查：另一人审查（重点看 AI 生成部分）
4. 合入主分支

审查重点：
- AI 生成的代码是否理解了每一行？
- 是否有隐含的安全/性能问题？
- 是否与项目现有代码风格一致？
```

#### 冲突避免 🔴

| 场景 | 风险 | 规范 |
|------|------|------|
| 两人同时用 AI 开发相关模块 | AI 生成的接口定义不一致 | 先定义接口文档，再各自实现 |
| AI 生成重复的工具类 | 两人各自让 AI 写了 StringUtils | 共享 utils 模块，使用前先搜索 |
| AI 生成的配置冲突 | 两人各自修改配置文件 | 配置变更必须同步 |

#### 小团队模板文件

```
# .cursorrules（2-3人团队精简版）

## 项目信息
- 技术栈：[填写]
- 代码风格：[填写]

## AI 使用规范
- 生成代码必须标注 [AI-Generated/Assisted/Reviewed]
- 提交前必须运行测试
- 每个功能最多让 AI 生成 3 次，不行就人工写

## 共享约定
- 接口变更必须先更新 API 文档
- 工具类放在 common/utils/ 下，新建前先搜索
- 配置变更必须在群里同步
```

## 三、中型团队（5～10 人）🔴🔴🔴

> 特点：开始有分工、需要流程、冲突概率显著上升

### 3.1 核心风险 🔴🔴

| 风险 | 表现 | 后果 |
|------|------|------|
| 冲突频发 | 多人同时用 AI 修改同一模块，Merge Conflict 爆炸 | 效率下降 |
| 上下文不一致 | 前端 AI 不知道后端 AI 生成的接口 | 集成失败 |
| 质量参差 | 有人 Prompt 写得好产出高，有人乱写产出差 | 代码质量不均 |
| 依赖爆炸 | AI 各自引入不同的第三方库 | 依赖冲突、安全风险 |
| 知识孤岛 | 好的 Prompt 和技巧不共享 | 团队效率上不去 |

### 3.2 规范 🔴🔴🔴

#### 分层分工 🔴🔴

```
团队分工：
- 技术经理：制定 AI 规范、审查质量、审批依赖
- 架构师：维护 .cursorrules、接口定义、核心模块不用 AI
- 开发工程师：AI 辅助功能开发、单元测试
- 测试工程师：AI 辅助测试用例生成、自动化测试
```

#### 冲突避免策略 🔴🔴🔴

| 策略 | 具体做法 |
|------|---------|
| **模块责任到人** | 每个模块指定唯一负责人，他人修改需通知 |
| **接口先行** | 前后端先定接口文档（可 AI 辅助生成），再各自实现 |
| **AI 编码分区** | 同一文件同一时间只允许一人用 AI 修改 |
| **统一依赖管理** | 新增依赖必须审批，AI 禁止自行引入未审批库 |
| **每日集成** | 每天至少一次集成，避免长期分支偏移 |

#### 代码审查规范 🔴🔴🔴

```
审查流程：
1. AI 先审：使用 CodeRabbit / PR-Agent 自动审查
2. 人工再审：指定审查人，重点审查 AI 生成部分
3. 审查要点：
   - [ ] AI 生成代码是否理解每一行
   - [ ] 是否有安全风险（注入、硬编码、越权）
   - [ ] 是否引入未审批依赖
   - [ ] 是否符合项目代码风格
   - [ ] 测试是否覆盖核心场景
   - [ ] 是否标注 AI 参与程度
```

#### 上下文管理 🔴🔴

```
项目上下文文件结构：
.cursorrules              # 全局规范
CLAUDE.md                 # AI Agent 行为指引
.ai/
  rules/
    security.md           # 安全规范
    coding-style.md       # 代码风格
    api-convention.md     # API 约定
    db-convention.md      # 数据库约定
  prompts/
    feature-dev.md        # 功能开发 Prompt 模板
    bug-fix.md            # Bug 修复 Prompt 模板
    code-review.md        # 代码审查 Prompt 模板
    test-gen.md           # 测试生成 Prompt 模板
  context/
    architecture.md       # 系统架构说明
    module-map.md         # 模块职责划分
```

#### Prompt 共享机制 🔴🔴

```
1. 建立 Prompt 共享文档（Notion/飞书）
2. 每周分享 1-2 个高效 Prompt
3. 新人入职必须学习团队 Prompt 模板
4. Prompt 模板定期迭代优化
```

#### 依赖管理 🔴🔴

```
新增依赖审批流程：
1. AI 生成代码时引入新依赖 → 标注在 PR 中
2. 审查人确认依赖必要性 + 安全性 + 许可证
3. 技术经理审批
4. 加入项目依赖白名单
5. 更新 .cursorrules 中的"已审批依赖"列表
```

## 四、大型团队（10～20 人）🔴🔴🔴🔴

> 特点：跨组协作、流程化要求高、失控风险最大

### 4.1 核心风险 🔴🔴🔴

| 风险 | 表现 | 后果 |
|------|------|------|
| 全面失控 | 各组 AI 使用方式不同，产出质量天差地别 | 系统质量不均 |
| 跨组冲突 | A 组 AI 生成的接口与 B 组不一致 | 集成灾难 |
| 安全事故 | 某人将生产数据输入 AI | 数据泄露 |
| 上下文碎片化 | 每组维护各自的 AI 上下文，全局不一致 | 协作困难 |
| 效率不均 | 有些组 AI 用得好效率翻倍，有些组几乎不用 | 资源浪费 |
| 合规风险 | AI 生成代码的版权/许可问题 | 法律风险 |

### 4.2 组织架构 🔴🔴🔴

```
AI 编程治理架构：

技术经理
  ├── AI 编程 Champion（1人）
  │     ├── 制定和维护 AI 编程规范
  │     ├── 培训和指导团队
  │     ├── 评估 AI 工具和效果
  │     └── 处理 AI 相关安全事件
  │
  ├── 小组 A（5-8人）
  │     ├── 小组 AI 负责人（兼）
  │     └── 开发人员
  │
  ├── 小组 B（5-8人）
  │     ├── 小组 AI 负责人（兼）
  │     └── 开发人员
  │
  └── 质量保障（2-3人）
        ├── AI 代码审计
        └── 安全审查
```

### 4.3 规范 🔴🔴🔴🔴

#### 分层治理 🔴🔴🔴

| 层级 | 职责 | 规范 |
|------|------|------|
| **全局层** | 技术经理 + AI Champion | 统一规范、工具选型、安全红线、依赖白名单 |
| **小组层** | 小组 AI 负责人 | 组内上下文维护、Prompt 优化、代码审查 |
| **个人层** | 每个开发人员 | 遵守规范、标注 AI 参与、审查 AI 输出 |

#### 冲突避免策略 🔴🔴🔴

| 策略 | 具体做法 | 负责人 |
|------|---------|--------|
| **模块所有权** | 每个模块指定 owner，修改需 owner 审查 | 技术经理 |
| **接口契约** | 跨组接口必须先定契约（OpenAPI/Proto），AI 只能按契约生成 | 架构师 |
| **集成频率** | 每日集成 + 每周全量集成测试 | 质量保障 |
| **变更通知** | 模块接口变更必须发通知 + 兼容期 | 小组 AI 负责人 |
| **冲突预处理** | AI 生成代码前先 git pull + rebase | 个人 |
| **统一代码生成模板** | 全团队使用相同的脚手架和模板 | AI Champion |

#### 代码审查体系 🔴🔴🔴

```
三层审查：

第一层：AI 自动审查（PR-Agent / CodeRabbit）
  → 自动检查：风格、安全、依赖、重复代码

第二层：组内审查（小组 AI 负责人指定审查人）
  → 重点检查：业务逻辑、AI 生成部分、接口兼容性

第三层：关键模块审查（架构师/技术经理）
  → 必须人工审查的模块：
    - 认证鉴权模块
    - 支付交易模块
    - 数据加密模块
    - 跨组接口定义
    - 核心业务逻辑
```

#### 上下文治理 🔴🔴🔴

```
全局上下文（全团队共享）：
├── .cursorrules              # 全局规范
├── CLAUDE.md                 # AI Agent 行为指引
├── .ai/
│   ├── rules/
│   │   ├── security.md       # 全局安全规范
│   │   ├── coding-style.md   # 全局代码风格
│   │   ├── api-convention.md # 全局 API 约定
│   │   ├── db-convention.md  # 全局数据库约定
│   │   └── dependency.md     # 依赖白名单
│   ├── prompts/              # 全局 Prompt 模板
│   └── context/
│       ├── architecture.md   # 系统架构
│       ├── module-map.md     # 模块职责+owner
│       └── interface-registry.md  # 跨组接口注册

小组上下文（组内共享）：
├── modules/[group-name]/
│   ├── .cursorrules          # 组内补充规范
│   ├── prompts/              # 组内 Prompt 模板
│   └── context/
│       ├── module-detail.md  # 模块详细设计
│       └── known-issues.md   # 已知问题和注意事项
```

#### 安全合规 🔴🔴🔴

```
安全审查清单（每两周一次）：
1. [ ] AI 生成代码的安全审查覆盖率
2. [ ] 敏感数据输入 AI 的事件数（应为 0）
3. [ ] 未审批依赖的引入情况
4. [ ] AI 生成代码的 Bug 密度
5. [ ] 关键模块 AI 参与度（核心模块应 < 30%）
6. [ ] AI 工具使用合规性（是否使用企业版）
```

#### 效率度量 🔴🔴

```
月度 AI 编程效率报告：
1. AI 工具使用率（团队/组/个人）
2. 代码产出变化（对比 AI 使用前后）
3. Bug 密度变化
4. 代码审查通过率
5. AI 生成代码占比
6. Prompt 模板使用情况
7. AI 相关安全事故数
```

## 五、三种团队对比速查

| 维度 | 2-3人 | 5-10人 | 10-20人 |
|------|-------|--------|---------|
| **核心风险** | 风格不统一、审查缺失 | 冲突频发、质量参差 | 全面失控、安全事故 |
| **AI Champion** | 技术经理兼 | 指定1人 | 专职1人+各组兼职 |
| **上下文管理** | 单一 .cursorrules | 分层上下文 | 全局+组内分层 |
| **代码审查** | 互相审查 | AI审+人审 | 三层审查体系 |
| **依赖管理** | 口头约定 | 审批流程 | 白名单+定期审计 |
| **冲突策略** | 接口先行+模块到人 | 模块所有权+每日集成 | 模块所有权+接口契约+三层审查 |
| **安全审查** | 每月1次 | 每两周1次 | 每周1次+关键模块实时 |
| **培训频率** | 启动时1次 | 每月1次 | 每两周1次+新人专项 |
| **度量周期** | 不需要 | 月度 | 周报+月报 |
| **核心原则** | 统一标准、互相审查 | 流程化、防冲突 | 分层治理、合规安全 |

## 六、推广路线图 🔴🔴

### Phase 1：试点期（第1-2周）

- [ ] 选 1-2 个非核心模块做 AI 编程试点
- [ ] 制定 `.cursorrules` 初版
- [ ] 培训 Prompt 编写规范
- [ ] 收集试点数据（效率、质量、问题）

### Phase 2：扩展期（第3-4周）

- [ ] 扩展到 50% 的开发任务
- [ ] 建立代码审查规范
- [ ] 共享高效 Prompt 模板
- [ ] 解决试点中发现的问题

### Phase 3：全面推广（第5-8周）

- [ ] 全团队使用 AI 编程
- [ ] 完善上下文管理体系
- [ ] 建立度量机制
- [ ] 形成团队 AI 编程知识库

### Phase 4：优化期（第9周+）

- [ ] 持续优化 Prompt 和流程
- [ ] 探索 Agent 模式的深度应用
- [ ] 建立团队 AI 编程最佳实践库
- [ ] 定期评估工具和方案
